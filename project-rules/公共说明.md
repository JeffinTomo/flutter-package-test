## 命名与注释
1. 不得使用中文
2. 所有函数的输入输出必须有严格的注释或者 interface 说明

## css 部分
1. 方法遵守 BEM https://www.cythilya.tw/2018/06/05/css-methodologies/
2. 尽量使用 taiwind
3. 需要核对实现，避免过多的冗余
4. svg 文件需要单独定义好再引用，不得直接放在代码逻辑中
5. 布局不得使用 absolute

## typescript 部分
1. 不允许出现 any
2. interface 的定义要放入单独的文件，最好是全局的统一文件

## 模板部分
1. 注意抽象 component
2. 模块化拆分，模块在独立的前提下，尽量小
3. 控制到 400 行以内 
4. 重要业务模块，请定义唯一且不变的 id, 命名要突出业务本身

## API 与数据
参考 ./api-hooks
1. api 封装成 service，并通过 hooks 暴露给业务层
2. 常用交互事件也封装 hooks

## sdk 开发
1. 在执行前要 严格 review 开发说明，指出问题或者不明确的地方
2. 基于结果自我审查，并输出执行过程，务必注明引用到的 lib
3. 务必输出开发使用手册，严格到 sdk 根目录的 cookbook.md 文件

## 其他
1. 遇到错误就 throw err
2. 遇到不确定的问题，可以分析对比后找我请求，不要瞎猜，更不可乱改
3. 执行任务的时，聚焦于任务本身，不要发散
4. review 时切换到一个与生成不同的模型


## 禁止事项
1. 不得违反 SOLID 与 DRY 原则，必须保持职责单一并避免逻辑重复
2. 不得维护复杂的中间状态，仅允许保留最小必要的核心数据
3. 不得保留未被使用的变量和函数
4. 不得在核心业务逻辑中混入 IO、网络、数据库等副作用操作
5. 不得形成隐式依赖，如依赖调用顺序、全局初始化或副作用时序
6. 不得臆测接口行为，必须先查询文档、定义或源码
7. 不得在缺乏上下文理解的情况下直接修改代码，必须基于整体结构审慎重构
8. 必须假设依赖库为权威且不可修改的黑箱实现， 不得对依赖库进行任何形式的功能裁剪、逻辑重写或降级封装
